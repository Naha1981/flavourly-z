## Phase 8 — PayFast Subscription Billing
**Goal:** Tenants can upgrade from trial to an active recurring monthly subscription, with production-grade signature validation.  
**Time estimate:** 3–4 hours  
**Read first:** `PAYFAST_INTEGRATION.md` — this phase implements what's documented there. Don't skip it; PayFast's signature scheme has a sharp edge (two different algorithms depending on which product you're calling) and the doc explains exactly which one applies here.

### Step 8.1 — Add the `payment_transactions` Table

Run this in Supabase SQL Editor (also included in the main schema block in `PLAN.md §2.1` if you're running this project fresh — skip if you already ran the full schema):

```sql
create table if not exists payment_transactions (
  id             uuid primary key default uuid_generate_v4(),
  tenant_id      uuid references tenants(id) on delete cascade,
  plan           text check (plan in ('starter','growth')),
  amount         numeric(10,2) not null,
  m_payment_id   text not null,
  pf_payment_id  text,
  payfast_token  text,
  status         text default 'pending'
                 check (status in ('pending','complete','failed','cancelled')),
  raw_itn        jsonb,
  created_at     timestamptz default now(),
  updated_at     timestamptz default now()
);
create index if not exists idx_payment_tx_tenant on payment_transactions(tenant_id);
create unique index if not exists idx_payment_tx_mpid on payment_transactions(m_payment_id);

alter table payment_transactions enable row level security;

create policy "payment_tx_tenant_read" on payment_transactions for select using (
  tenant_id in (select tenant_id from profiles where id = auth.uid())
  or exists (select 1 from profiles where id = auth.uid() and role = 'super_admin')
);
```

This table is what makes the ITN handler idempotent and tamper-resistant — without it you can't verify the amount PayFast confirms actually matches what you charged.

### Step 8.2 — Get Your PayFast Sandbox Credentials

1. Sign up at [sandbox.payfast.co.za](https://sandbox.payfast.co.za) if you haven't
2. Note your sandbox Merchant ID and Merchant Key from the dashboard
3. **Set a passphrase** — go to Settings → Security Passphrase. This is mandatory: recurring billing will not work without one, and PayFast will reject subscription checkouts silently if it's missing.
4. Sandbox and production have **separate** credentials — never reuse sandbox values in production secrets.

### Step 8.3 — Set Supabase Secrets

```bash
supabase secrets set \
  PAYFAST_MERCHANT_ID="[sandbox-merchant-id]" \
  PAYFAST_MERCHANT_KEY="[sandbox-merchant-key]" \
  PAYFAST_PASSPHRASE="[your-sandbox-passphrase]" \
  PAYFAST_MODE="sandbox"
```

`PAYFAST_MODE` is what both functions read to pick between `sandbox.payfast.co.za` and `www.payfast.co.za` — this is the only thing you change when going live (Step 8.9).

### Step 8.4 — Deploy Both PayFast Functions

```bash
mkdir -p supabase/functions/payfast-checkout
mkdir -p supabase/functions/payfast-itn

cp payfast-checkout.ts supabase/functions/payfast-checkout/index.ts
cp payfast-itn.ts       supabase/functions/payfast-itn/index.ts

supabase functions deploy payfast-checkout
supabase functions deploy payfast-itn --no-verify-jwt
```

`payfast-checkout` requires auth (a logged-in tenant calls it). `payfast-itn` does not — PayFast calls it directly with no Supabase auth token, so `--no-verify-jwt` is required or every ITN will be rejected with a 401.

### Step 8.5 — Confirm the ITN Endpoint Is Publicly Reachable

PayFast cannot reach an endpoint that requires auth or is behind a firewall. Verify:

```bash

  -d "test=1"
```

You should get a response (even an error response is fine — a timeout or connection refused means PayFast will never be able to notify you of a payment).

### Step 8.8 — Test the Security Checks Deliberately

Don't skip this — it's the difference between a working integration and a silent vulnerability:

| Test | How | Expected result |
|---|---|---|
| Tampered amount | In `payment_transactions`, manually change `amount` to a different value right after clicking Upgrade but before completing sandbox payment | ITN should log an amount mismatch and set `status = 'failed'`, tenant stays on trial |
| Wrong source IP | Manually POST to your `payfast-itn` URL from your own machine with a fabricated but correctly-signed payload | Should be rejected at the IP check — confirm via function logs |
| Bad signature | POST to `payfast-itn` with a random `signature` value | Should be rejected immediately, no DB writes |
| Repeat ITN | Re-send the same successful ITN payload a second time (PayFast sandbox lets you resend from their dashboard) | Should log "repeat charge confirmed" and not error, not double-process |

Check function logs for all of these: `supabase functions logs payfast-itn`
